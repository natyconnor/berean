Fade-Dial Learning Model

Turn "learning" a verse from one-exact-per-rung into a rep-gated support dial that fades from full hints to none in a single session, leaving the 30-day learned-to-mastered spaced-repetition arc intact.

5 pull requests no new runtime deps one Convex schema field

Do not start the next PR until the current one is green: implement only what that PR lists, run pnpm run agent:check (and pnpm test where the PR touches a *.test.ts), satisfy the PR's acceptance criteria, then stop.
Settled decisions

Decisions locked before implementation

Fixed constraints resolved with the product owner. Treat as non-negotiable.

Decision Resolution
Learning is one session new → learned is the whole support dial, done within a session. Reaching zero support and landing the from-memory rep graduates the verse to reviewing (= "learned").
Learned → mastered unchanged The existing reviewing-phase interval growth is untouched; a verse becomes mastered once its interval reaches MASTERED_INTERVAL_DAYS (30). Reviews are always from memory.
Discrete bands, increasing reps Four bands map 1:1 onto today's learnStage 0..3. Required exact reps per band: Read 1, Guided 5, Challenge 8, From Memory 1 (graduate). Tunable via one config array.
Miss handling close repeats the same band (no rep banked); off bumps support up one band and resets that band's rep counter.
Progress persists A new stageReps field is saved on every attempt so leaving mid-band resumes in place rather than restarting.
Hints re-randomize and fade Each rep reseeds which words show a first-letter hint; within the Challenge band the hint density fades across its reps ("fewer and fewer first letters").
Naming Retire mechanism labels (Full/Letters/Blanks/Hidden) for Read → Guided → Challenge → From Memory. "Mastery" is deliberately avoided as a level name so it stops colliding with the mastered status.

Shared architecture

Data model & core contracts (referenced by every PR)

Support bands (single source of truth, in the scheduler)

// src/lib/memory-scheduler.ts — index === learnStage (0..3)
export interface SupportBand {
  key: "read" | "guided" | "challenge" | "memory";
  label: string;              // "Read" | "Guided" | "Challenge" | "From Memory"
  requiredReps: number;       // exact reps to clear this band
  densityStart: number | null; // first-letter hint fraction at rep 0; null = full text
  densityEnd: number | null;   // hint fraction at the last rep (fades within a band)
}

export const SUPPORT_BANDS: readonly SupportBand[] = [
  { key: "read",      label: "Read",        requiredReps: 1, densityStart: null, densityEnd: null },
  { key: "guided",    label: "Guided",      requiredReps: 5, densityStart: 1.0,  densityEnd: 1.0  },
  { key: "challenge", label: "Challenge",   requiredReps: 8, densityStart: 0.5,  densityEnd: 0.15 },
  { key: "memory",    label: "From Memory", requiredReps: 1, densityStart: 0.0,  densityEnd: 0.0  },
];
export const MAX_LEARN_STAGE = 3; // unchanged

MemorySchedule (one new field: stageReps)

export interface MemorySchedule {
  status: MemoryStatus;       // "new" | "learning" | "reviewing" | "mastered"
  learnStage: number;         // 0..3 band index (unchanged meaning)

  ease: number;
  intervalDays: number;
  dueAt: number;
  consecutiveCorrect: number;
  lapses: number;

Learning-phase transition (rep-gated)

// scheduleLearning(s, r): only the learning branch changes.
if (r.quality === "exact") {
  const reps = s.stageReps + 1;
  if (reps >= SUPPORT_BANDS[s.learnStage].requiredReps) {
    if (s.learnStage >= MAX_LEARN_STAGE) {
      // graduate: reviewing, interval 1d (existing computeDueAt), stageReps 0
    } else {
      // advance band: learnStage+1, stageReps 0, intervalDays 0, dueAt now
    }
  } else {
    // bank a rep: same learnStage, stageReps = reps, intervalDays 0, dueAt now
  }
} else if (r.quality === "close") {
  // hold: same learnStage, stageReps unchanged, dueAt now
} else { // "off"
  // bump up: learnStage = max(0, learnStage-1), stageReps 0, consecutiveCorrect 0
}

Data flow

A learner opens Practice or the Review learn card, which restores the verse's server-authoritative learnStage + stageReps (via getOrCreateForVerse / the hearted-verses list). The card calls hintForProgress(learnStage, stageReps) and feeds the result into maskVerseText to render the current rung of the dial. The learner types and checks; classifyVerseAttempt grades the diff as exact | close | off; record() calls verseMemory.recordAttempt, which logs a verseMemoryReviews row and runs the pure scheduleNext. The returned schedule (now including stageReps) is persisted and adopted by the UI, advancing the dial, banking a rep, holding, or bumping support up. When the from-memory rep at the last band lands exact, the verse graduates to reviewing ("learned") and the existing interval-growth path carries it to mastered.

Rules for the agent

Conventions every PR must follow

Workflow

  • One PR at a time, in order. Read files before editing.
  • After code changes run pnpm run agent:check (format → lint → typecheck).
  • Run pnpm test for any PR that adds/edits a *.test.ts (PRs 1 and 2 at minimum).
  • Run pnpm run codegen after the schema/validator change in PR 1 so convex/_generated stays in sync.
  • Make the smallest correct change; preserve existing behavior not named by the PR. Do not revert unrelated working-tree changes.
  • Never commit or push unless explicitly asked.
  • When the model changes, update docs/study-mode.md (done in PR 5).

Domain rules (Convex + TS, enforced by lint/typecheck)

  • Every public query/mutation keeps args and returns validators; add stageReps to the relevant validators, not just the types.
  • Preserve auth/ownership checks already present; do not weaken getCurrentUser/ownership guards.
  • Never call Date.now() inside a query — now is passed in. The scheduler stays pure (no Convex, no React, no Date.now()).
  • Query with .withIndex(...), never .filter(...); await every promise.
  • No any; use generated Doc / Id types. Read legacy rows defensively (row.stageReps ?? 0).
  • stageReps is schema-optional (backfill safety) but always written going forward.

Fading, re-randomized hints

Depends on: PR 1 (for SUPPORT_BANDS). Unblocks: PR 3, 4.

Goal

Make masking parameterizable so each rep shows a different, progressively sparser set of first-letter hints, and add the shared hintForProgress mapping both learn surfaces will use. No UI wiring yet, so nothing visibly changes.

Files

  • src/lib/verse-hint.ts — add the MaskOptions (density, seed) parameter to maskVerseText; fold both into getClozeFirstLetterIndices so the hinted subset is chosen by density and reshuffled by seed; add and export hintForProgress(learnStage, stageReps).
  • src/lib/verse-hint.test.ts — add coverage for density + seed behavior and hintForProgress; keep the existing full/first-letters/cloze/hidden cases green (default behavior with no options must be unchanged).

Implementation steps

  • Keep the current call shape working: maskVerseText(text, stage) with no options behaves exactly as today (cloze stays deterministic on verse text).
  • When density/seed are supplied for cloze, select roughly density * wordCount first-letter hints, ranking words by a hash of (text, index, seed) so a new seed yields a different subset while a repeated seed is deterministic.
  • Implement hintForProgress from SUPPORT_BANDS: read → full; guided → first-letters, density 1; challenge → cloze with density = lerp(densityStart, densityEnd, stageReps/(requiredReps-1)) and seed = stageReps; memory → hidden.
  • Run pnpm run agent:check and pnpm test.

Acceptance criteria

  • pnpm run agent:check and pnpm test pass; existing maskVerseText tests remain green unchanged.
  • Tests prove: two different seeds at the same density produce different hinted subsets, each deterministic for a fixed seed; a lower density yields fewer first-letter hints; every word token is still masked (never the plain word) and punctuation/spacing is preserved.
  • hintForProgress returns the expected stage/density/seed for representative (learnStage, stageReps) inputs across all four bands.

Risks & how the plan answers them

Risk Mitigation (baked into the PR sequence)
Legacy verseMemory rows lack stageReps. Field is v.optional and read as ?? 0 everywhere; a mid-ladder learner simply starts the current band's reps at 0 (PR 1).
Scheduler change also affects the Review learn phase. Intended (one model). Both surfaces are updated (PR 3, PR 4) and the reviewing-phase deck stays from-memory and untouched.
Read prime has no natural graded attempt to advance the server band. The Continue action submits the shown full text as the single Read rep, so the server stays authoritative with no new mutation (PR 3, PR 4).
Grindy feel from 5 + 8 reps in one sitting. Rep counts live in one SUPPORT_BANDS array and are trivially tunable; hints re-randomize and fade so reps do not feel identical (PR 1, PR 2).
Two progress vocabularies confuse the UX. Practice's manual stage selector becomes progress-only; status stays the one lifecycle concept, surfaced consistently in the ring/labels (PR 3, PR 5).
Validator/type drift across the Convex surface. PR 1 threads stageReps through every validator and runs pnpm run codegen; agent:check gates each PR.
